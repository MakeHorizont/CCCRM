
# Кодекс Разработчика: Товарищеская Критика

Мы строим систему коллективного пользования, поэтому и процесс ее создания должен быть коллективным и осознанным.

## 1. Принципы Кода
*   **Чистота:** Код — это литература для машин и людей. Пиши так, чтобы товарищ, который придет после тебя, понял твою мысль без расшифровки.
*   **Строгость:** TypeScript включен не для галочки. `any` — это оппортунизм. Используй строгие типы.
*   **Материализм:** Любая функция должна иметь связь с реальностью. Не пиши абстракции ради абстракций.

## 2. Процесс Внесения Изменений
1.  **Форк и Ветка:** Создай форк репозитория. Ветку назови по сути задачи: `feature/add-audit-log` или `fix/salary-calculation`.
2.  **Коммиты:** Пиши осмысленные сообщения коммитов.
    *   *Плохо:* `fix bug`
    *   *Хорошо:* `fix: исправлен расчет КТУ при неполной смене`
3.  **Pull Request (PR):**
    *   В описании PR объясни, **зачем** это изменение нужно коллективу.
    *   Приложи скриншоты, если менял интерфейс.
    *   Убедись, что тесты проходят (если они есть).

## 3. Товарищеская Критика (Code Review)
Мы не "надзираем", мы помогаем сделать лучше.
*   **Критикуй код, а не автора.**
*   Предлагай альтернативы, а не просто указывай на ошибки.
*   Хвали за удачные решения.

## 4. Структура Проекта
*   `/components` — Визуальные компоненты (станки).
*   `/services` — Бизнес-логика и API (цеха).
*   `/types` — Определения типов (чертежи).
*   `/documents` — Документация и планы (Госплан).

## 5. Идеологическая Безопасность
Перед тем как добавить фичу, проверь её по чеклисту `documents/analysis/ideological_security.md`:
1.  Не создает ли она возможность для эксплуатации?
2.  Не нарушает ли она прозрачность?
3.  Способствует ли она сплочению коллектива?
